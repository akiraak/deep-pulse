# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

deep-pulse はニュースを深く考察した記事を Markdown として生成し、HTML サーバーで配信するツール。
記事の生成は Claude Code 上で行い、`output/` ディレクトリに保存する。

## 作業の進め方

- 実装タスクを依頼された場合、まずプラン（計画）を作成してユーザーの確認を取ること
- **ユーザーが明示的に実装を承認するまでコードの変更・ファイルの作成を行わない**
- 「プランを作成して」と言われた場合は、プランファイルの作成のみを行い、実装には進まない
- プランは `plans/` ディレクトリに Markdown ファイルとして保存する（例: `plans/feature-name.md`）
- プラン作成後、`TODO.md` の該当タスクにプランファイルへのリンクを追記する（例: `[プラン](plans/feature-name.md)`）

## Commands

```bash
npm install          # 依存パッケージのインストール
npm run serve        # HTML サーバー起動 (http://localhost:3000)
npm run build        # TypeScript コンパイル
npm start            # コンパイル済みコードの実行
npm run build:site   # 静的サイトビルド (dist_site/ に出力)
npm run lint         # ESLint
npm run format       # Prettier
```

## Architecture

- `src/index.ts` — エントリーポイント。サーバーを起動する
- `src/render.ts` — Markdown → HTML 変換（marked 使用）。OGP タグ・ヘッダー・パンくず生成を含む
- `src/server.ts` — HTTP サーバー。記事一覧・個別記事を HTML で配信
- `scripts/build_static.ts` — 静的サイトビルド。GitHub Pages 用。`ogp.png` も `dist_site/` にコピーする
- `.github/workflows/pages.yml` — push 時に GitHub Pages へ自動デプロイ

## Article Generation Rules

記事は Claude Code がこの画面上で生成し、`output/` に保存する。以下の手順で進める。

### ディレクトリ構成

各記事は `output/` 内に専用ディレクトリを作成して管理する:

```
output/
  YYYY-MM-DD_タイトル/
    YYYY-MM-DD_タイトル.md          # 記事本体
    YYYY-MM-DD_タイトル_plan.md     # 生成プラン
    YYYY-MM-DD_タイトル_note.md     # note用紹介記事（記事一覧には含めない）
    sources/                         # 取得した HTML ソース
      001_ドメイン名_ページ概要.html
      002_ドメイン名_ページ概要.html
      ...
```

- `sources/` ディレクトリには、情報収集時に WebFetch で取得した各 URL のコンテンツを Markdown 形式で保存する
- **ソースは要約せず、元記事の文章をそのまま保存すること**（記事生成時にオリジナルの文章で検証するため）
- ファイル名は `連番_ドメイン名_ページ概要.md` の形式とする（例: `001_nikkei_バター卸値上昇.md`）
- **各ソースファイルの先頭には以下のメタ情報を YAML フロントマターで記載すること:**
  ```
  ---
  url: https://example.com/article
  title: 元記事のタイトル
  ---
  ```
  - `title` は元ページの `<title>` タグやページ見出しから取得する（リンクテキストに使用するため正確に記録する）
- **ソース取得には `scripts/fetch_source.py` を使うこと**（WebFetch ではなく curl + pandoc で原文を保持する）:
  ```bash
  python3 scripts/fetch_source.py <url> <output_path>
  ```
  - 複数 URL を並列取得する場合はバックグラウンド実行 (`&`) と `wait` を組み合わせる
  - スクリプトが自動的に YAML フロントマター（url, title）を付与する
- プランファイルのソース URL リストには、対応する `sources/` 内のファイル名も併記する

### 手順

1. **テーマ確認** — ユーザーが記事のテーマ・時期・トピックを指定する
2. **情報収集** — WebSearch / WebFetch を使い、関連するニュース記事・一次情報をできる限り多く収集する。**取得した各 URL の HTML コンテンツは `sources/` ディレクトリに保存する**
3. **生成プラン作成** — 収集した情報を元に記事の構成案を `YYYY-MM-DD_タイトル_plan.md` として保存し、ユーザーに確認を取る。プランファイルには以下を含めること:
   - **`## ユーザー指示`** セクション — ユーザーが指定したテーマ・指示テキストをそのまま原文で記録する
   - 収集した全ソースの URL を**各セクションごとに**記録する（コンテキスト圧縮で失われないようにするため）
   - 各ソース URL に対応する `sources/` 内のファイル名を併記する
4. **記事生成** — プランに基づき `YYYY-MM-DD_タイトル.md` として記事を生成する。**各セクションを書く前に、`sources/` に保存された該当ソースのファイルを読み込み、内容を確認しながら執筆すること**
5. **ファクトチェック** — 生成した記事の各セクションについて、参照した `sources/` 内のファイルと突き合わせ、事実・数値・引用に齟齬がないか検証する。**齟齬が見つかった場合は記事を修正し、齟齬がなくなるまでこのステップを繰り返す**
6. **note用紹介記事生成** — 元記事から `YYYY-MM-DD_タイトル_note.md` を生成する。元記事への誘導を目的とした短い紹介文。構成は以下の通り:
   - **キャッチーなタイトル** — 元記事と同じか、より引きのあるタイトル
   - **リード文（2〜3文）** — 読者の関心を掴むフック。問いかけや意外な事実から入り、記事の内容が分かりやすく伝わるようにする
   - **元記事リンク** — リード文の後に `> 元記事（グラフ・相関図付き）はこちら: URL` を配置する。URL は `https://akiraak.github.io/deep-pulse/articles/{記事ファイル名}.html`
   - **見どころ紹介（箇条書き3〜5項目）** — 記事の読みどころを端的に。見出しは「読みどころ」など、簡潔で自然な表現にする
   - **文章量は元記事の1/10以下**を目安にコンパクトにまとめる
   - Mermaid・Chart.js・テーブル・参考ソースは含めない
7. **静的サイトビルド** — 記事生成後、`npm run build:site` を実行して HTML を生成する。`dist_site/` に記事の HTML とトップページ（`index.html`）のリンクが正しく出力されていることを確認する
8. **コミット** — 記事の Markdown（`output/`）をコミットする（`dist_site/` は `.gitignore` に含まれるため、コミット対象外。GitHub Pages デプロイは CI が `npm run build:site` を実行して行う）。`sources/` ディレクトリもコミット対象とする

### 記事のフォーマット

- ファイル名: `YYYY-MM-DD_タイトル.md`（日付は記事の生成日）
- 記事は日本語で執筆する
- 以下の観点で深い考察を加えること:
  - 事実の整理と背景の説明
  - 複数の視点からの分析
  - 社会的・経済的影響の考察
  - 今後の展望
- Markdown の見出し・箇条書き・引用を適切に使用する
- **参考ソースの表示形式:**
  - 各セクション（`##` 見出し）の末尾に「**参考ソース:**」として、そのセクションで参照した記事をリスト化する
  - リンクは `sources/` 内のファイルのフロントマターから取得した `title` をリンクテキストに使い、`url` をリンク先にする: `- [記事タイトル](URL)`
  - 記事末尾の「## 参考ソース」セクションは不要（各セクション末尾に分散配置するため）

### 記事のスタイルガイド

- **キャッチーで分かりやすい文体** — 難しいテーマでも具体例・例え話を多用して読者の理解を助ける
- **登場人物が多いテーマの場合** — 記事冒頭に「登場人物と相関図」セクションを設ける:
  - 主要関係者を**表（テーブル）**で整理する（人物・立場・役割の3列）
  - 関係性を**Mermaid フローチャート**（` ```mermaid ` コードブロック）で可視化する
  - 複数のレイヤー（例: 政治サイド / 運営サイド / 規制サイド）に分類すると見通しが良くなる
- **問題を分析するテーマの場合** — 「問題あり」と「問題なし」を明確に切り分ける構成にする
  - 各論点ごとに判定（問題あり / 問題なし）を明示する
  - まとめセクションで赤信号・青信号の表として一覧化する
- **表（テーブル）を積極的に活用する** — 比較、時系列、配分構造などは文章より表のほうが伝わる
- **例え話のルール** — 各セクションに最低1つ、日常生活に置き換えた例え話を入れる
- **本文中のインラインリンク** — セクション末尾の「参考ソース」リストに加えて、本文中の重要な事実・数値・キーワードにも参照元へのインラインリンクを付ける。読者がその場で出典を確認できるようにする
- **グラフ（Chart.js）の活用** — 数値の推移・比較・分布などは、テーブルに加えてグラフでも可視化する。Markdown 内で以下の記法を使う:
  - ` ```chart ` コードブロック内に Chart.js の設定オブジェクト（JSON）を記述する
  - レンダリング時に自動的に `<canvas>` + Chart.js グラフに変換される
  - **使い分け**: 折れ線グラフ（時系列推移）、棒グラフ（比較）、ドーナツ/円（構成比）を適切に選択する
  - **記法例**:
    ````
    ```chart
    {
      "type": "line",
      "data": {
        "labels": ["22歳", "30歳", "40歳", "50歳", "60歳", "65歳"],
        "datasets": [
          {
            "label": "手取り10%積立",
            "data": [0, 500, 1500, 3200, 5800, 7200],
            "borderColor": "#8b0000",
            "fill": false
          }
        ]
      },
      "options": {
        "plugins": {
          "title": { "display": true, "text": "資産推移シミュレーション（万円）" }
        }
      }
    }
    ```
    ````
  - 1 記事内に複数グラフ OK。テーブルとグラフはセットで使うと効果的（テーブルで正確な数値、グラフで傾向を直感的に把握）
- **図・相関図（Mermaid.js）の活用** — 人物相関図、タイムライン、フロー図などは Mermaid.js で描画する。Markdown 内で以下の記法を使う:
  - ` ```mermaid ` コードブロック内に Mermaid 記法を記述する
  - レンダリング時に自動的に SVG 図に変換される（サイトのカラースキームで描画）
  - **注意事項**:
    - サブグラフ名に日本語や特殊文字を含む場合は `subgraph id["表示名"]` 形式で書く
    - ノードテキストは `["テキスト"]` でクォートする
    - 改行は `\n` を使う
    - シンプルな直線フローを心がける（分岐が多いとレイアウトが崩れやすい）
  - **記法例**:
    ````
    ```mermaid
    flowchart TD
        A["1967年 創業\nRalph Ward & Bud Williams"] --> B["1985年 転換点\n従業員が会社を買収"]
        B --> C["ESOP設立\n従業員オーナー企業へ"]
        C --> D["現在 141店舗以上"]
    ```
    ````

### ルールの更新

- ユーザーから記事の生成方法・スタイル・構成について新たな指示があった場合は、その内容をこの `CLAUDE.md` の該当セクション（Article Generation Rules / 記事のスタイルガイド）に反映すること
- 既存ルールと矛盾する場合は、新しい指示で上書きする

## Task Management

- タスクは `TODO.md` で管理する
- コミット前に、完了したタスクを `TODO.md` から `DONE.md` に移動すること

## Conventions

- コメント・ドキュメント・生成記事は日本語
- 変数名・関数名・ファイル名は英語
- ES Modules (`"type": "module"`)、インポートには `.js` 拡張子を付ける
